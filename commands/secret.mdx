---
title: "podman secret — Store and manage sensitive runtime data"
description: "Reference for podman secret subcommands. Store passwords, API tokens, TLS certificates, and SSH keys for container access without embedding them in images."
---

Secrets are named blobs of sensitive data — passwords, TLS certificates, SSH keys, or other binary content — that containers need at runtime but should never be stored in an image or in source control. Podman stores secrets separately from containers and images. They are not included in `podman commit` or `podman export` archives, and they can be mounted into containers at a controlled path or exposed as environment variables. A single secret can be up to 512 kB in size.

## Subcommands

| Subcommand | Description |
|---|---|
| `create` | Create a new secret. |
| `exists` | Check if a named secret exists. |
| `inspect` | Display detailed information on one or more secrets. |
| `ls` | List all available secrets. |
| `rm` | Remove one or more secrets. |

## secret create

Creates a secret from a file path or from stdin. The secret name is required; the second argument is either a file path or `-` to read from stdin.

```
podman secret create [options] <name> <file|->
```

<Note>
Secret data is never printed by `podman secret inspect` or `podman secret ls`. Only metadata (name, ID, labels, driver) is shown.
</Note>

### Key flags

<AccordionGroup>
  <Accordion title="--driver / -d">
    Choose the storage driver for the secret. Three drivers are built in:

    | Driver | Description |
    |---|---|
    | `file` (default) | Stores the secret in a read-protected file on disk. |
    | `pass` | Stores the secret in a GPG-encrypted file (requires GPG). |
    | `shell` | Delegates storage to custom scripts configured in `containers.conf`. |

    ```bash
    # Store using the GPG pass driver
    podman secret create --driver pass mykey ./key.txt.gpg
    ```
  </Accordion>

  <Accordion title="--env">
    Read the secret value from an environment variable instead of a file. The second argument becomes the variable name.

    ```bash
    export DB_PASSWORD="s3cr3t"
    podman secret create --env true db_password DB_PASSWORD
    ```
  </Accordion>

  <Accordion title="--replace">
    If a secret with the same name already exists, overwrite it. The updated secret only affects newly created containers — existing running containers are not changed. Cannot be combined with `--ignore`.

    ```bash
    podman secret create --replace mytoken ./new-token.txt
    ```
  </Accordion>

  <Accordion title="--ignore">
    If a secret with the same name already exists, do not fail — return the existing secret's ID instead. Cannot be combined with `--replace`.

    ```bash
    podman secret create --ignore mytoken ./token.txt
    ```
  </Accordion>

  <Accordion title="--label / -l">
    Attach key-value metadata labels to the secret. Labels are visible in `podman secret inspect` and `podman secret ls`.

    ```bash
    podman secret create --label team=ops --label env=prod db_password ./password.txt
    ```
  </Accordion>
</AccordionGroup>

### Examples

```bash
# Create a secret from a file
echo -n "my-database-password" > ./dbpass.txt
podman secret create db_password ./dbpass.txt

# Create a secret from stdin (avoids writing to disk)
printf "my-database-password" | podman secret create db_password -

# Create a secret from an environment variable
export DB_PASS="s3cr3t"
podman secret create --env true db_password DB_PASS

# Generate a cryptographically random secret and store it
openssl rand -base64 32 | podman secret create --replace api_key -

# Create a secret using the GPG pass driver
podman secret create --driver pass tls_cert ./cert.pem.gpg

# Create a secret with labels
podman secret create --label app=myapp tls_key ./tls.key
```

## Using secrets in containers

Secrets are passed to containers via `--secret` at `podman run` or `podman create` time.

```bash
# Mount a secret as a file at /run/secrets/<name>
podman run --rm \
  --secret db_password \
  docker.io/library/alpine \
  cat /run/secrets/db_password

# Mount at a custom path with custom ownership and permissions
podman run --rm \
  --secret source=db_password,type=mount,target=/etc/app/password,uid=1001,gid=1001,mode=440 \
  my-app-image

# Expose a secret as an environment variable
podman run --rm \
  --secret source=db_password,type=env,target=DB_PASSWORD \
  my-app-image env | grep DB_PASSWORD
```

<Warning>
Secrets exposed as environment variables (`type=env`) are visible to all processes in the container and may appear in process listings on the host. Prefer file mounts (`type=mount`) for highly sensitive values.
</Warning>

## secret ls

List all secrets and their metadata. Secret values are never shown.

```bash
# List all secrets
podman secret ls

# Filter by label
podman secret ls --filter label=env=prod

# Show only secret IDs
podman secret ls -q
```

## secret inspect

Return JSON metadata for one or more secrets.

```bash
# Inspect a secret
podman secret inspect db_password

# Inspect multiple secrets
podman secret inspect db_password api_key

# Format output
podman secret inspect --format '{{.Spec.Name}} {{.ID}}' db_password
```

## secret rm

Remove one or more secrets. Removing a secret does not affect containers that are already running — only newly created containers are affected.

```bash
# Remove a single secret
podman secret rm db_password

# Remove multiple secrets
podman secret rm db_password api_key tls_cert

# Remove all secrets
podman secret ls -q | xargs podman secret rm
```

## Common workflows

```bash
# Rotate a secret without downtime
openssl rand -base64 32 | podman secret create --replace api_key -

# Use a secret for registry authentication (podman login)
echo -n "$REGISTRY_TOKEN" | podman secret create registry_token -
podman login --secret registry_token registry.example.com

# Pass a TLS certificate and key to a web server container
podman secret create tls_cert ./server.crt
podman secret create tls_key  ./server.key
podman run -d \
  --secret source=tls_cert,type=mount,target=/etc/nginx/ssl/server.crt \
  --secret source=tls_key,type=mount,target=/etc/nginx/ssl/server.key \
  -p 443:443 \
  docker.io/library/nginx:latest
```
